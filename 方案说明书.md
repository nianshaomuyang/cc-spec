1. 产品核心愿景与架构设计

1.1 一句话定位

一个常驻系统托盘、免 API 接入、全手动触发的 **Spec 迭代演进与终端会话打捞的副屏数字孪生大屏**。

1.2 核心三大设计哲学

1. **终端依然是唯一聊天主战场**：你想怎么和 Claude Code 聊就怎么聊，对 AI 的输出格式做零硬性格式约束（方案 B），工具对开发者带来零额外心智负担。
2. **纯手动触发，绝对安全与克制**：彻底抛弃后台静默扫描。只有当你明确下达命令或点击按钮时，工具才执行对应的动作。
3. **高维资产管理**：代码可以重新写，但 Spec 是项目的灵魂。工具的职责是维护 Spec 及其技术决策（ADR）在迭代过程中的绝对稳定与演进时光回溯。

2. 📂 项目目录契约规范 (Workspace Contract)

应用启动后，用户直接通过桌面端将本地项目文件夹拖入。软件通过后台常驻的本地文件监听引擎（如 `chokidar`），仅对以下约定的 `spec/` 目录进行单向只读监听：

text

```
your-project/
├── spec/                        # ClaudeSpec 资产真理源
│   ├── SPEC.md                  # 【MASTER】真理源主文档（包含你的模块规范 + 末尾追加的 Scratchpad 历史章节）
│   └── decisions/               # 【未决暂存区 / 方案对比区】
│       ├── auth_options.md      # Claude 随手生成的 Option 1/2/3 模糊方案（用于方案卡片渲染）
│       └── payment_stage.md     # 待评审的新 Spec 暂存文件（用于与主 SPEC.md 做跨文件 Diff 拦截）
└── src/                         # 业务源码区（仅用于监控变动频次，做温和文档滞后提醒）
```

3. 🛠️ 终极功能矩阵 (Feature Specifications)

3.1 核心 1：大文档 H2/H3 智能虚拟切片树 (Virtual Split Tree)

- **痛点**：随着项目迭代式开发，物理大单文件 `spec/SPEC.md` 会变成数千行的庞然大物，人类阅读极其痛苦，整篇喂给 AI 又极耗 Token 且容易引发 Lost in the Middle 幻觉。
- **解法**：保持项目唯一的物理大文件。桌面端内置 Markdown AST（抽象语法树）解析器，实时监听 `spec/SPEC.md`。只要文件一变，前端自动按 `##`（H2，模块名）或 `###`（H3，子条款/历史讨论章节）将其物理单文件切碎。
- **Web 端呈现**：左侧动态渲染出多层级的原生模块树导航，右侧点击对应节点以清爽的 Tab 标签页和美观的 Markdown 视图展现单章内容。

3.2 核心 2：方案 B 智能解耦决策墙 (Smart Option Matrix)

- **痛点**：终端推演的多技术方案（Option 1/2/3）随聊随走，不仅格式混乱，且极易遗忘两周前为什么放弃了方案 A 选了方案 B。
- **解法**：对终端聊天做零格式约束。只要内容被写入 `spec/decisions/` 目录，后端文本特征块分析器会利用模糊正则自动捕捉文本中的“逻辑断层”和“对比信号”（如捕捉 `Option A`、`方案一`、`优缺点`、`复杂度` 等信号）。
- **Web 端呈现**：在副屏将长篇大论自动揉碎，并排渲染成精美的**方案对比卡片墙**。卡片底部提供输入框，由用户手动输入【最终选择原因】，点击确认后，静默向该方案文件末尾写入并归档为长期可追溯的架构决策记录（ADR）。

3.3 核心 3：暂存区 Diff 强力拦截大屏 (Staging Diff Gatekeeper)

- **痛点**：用户弹出修改意见后，AI 局部短视听风就是雨，立刻去改代码，导致全局业务逻辑或边界条件被偷偷篡改或意外破坏，最终不得不频繁 Git Reset。
- **解法**：强制执行“图纸先行”工作流。你想迭代新需求时，命令 AI **先别动代码**，将新 Spec 输出至暂存区（如 `spec/decisions/staging_spec.md`）。
- **Web 端呈现**：桌面端一键调起 `react-diff-viewer`，直接拉取最新的主 `SPEC.md` 对应章节与该暂存文件进行**跨文件语义化红绿比对**。不仅显示废话的修改，更智能提取结构化业务规则的变动（如：旧规则 ➔ 新规则）。用户肉眼评审确认无误后，再切回终端解锁代码修改命令。

3.4 核心 4：命令驱动·会话章节追加打捞器 (Command-Driven Appender)

- **痛点**：在终端和 Claude Code 聊到极其精妙的底层技术推导、或者 AI 聊爆崩溃前的那段珍贵黄金共识，随着终端清屏或 Session 断线而永远消失，去翻零散的历史日志极其耗时。
- **解法**：提供一个常驻系统终端的轻量级 CLI 工具。当你觉得当前或最近一次对话极具价值时，在终端输入一行极简命令：

  ```bash
  cs-save “你的章节名称”
  ```

- **执行逻辑**：

  1. CLI 瞬间触发后端，物理读取 Claude Code 留在系统缓存路径中的最新一条会话 JSONL 日志。
  2. 精准截取最后一次 AI 的输出文本，将其格式化为标准的 Markdown 文本。
  3. **直接追加（Append）写入到本地项目主 `spec/SPEC.md` 的 `## Scratchpad（随手记）` 大章节最末尾，作为一个全新的 `### 你的章节名称` 子标题。**
  4. 触发主文档文件变动事件，桌面端的虚拟切片树”叮”的一声无感瞬间刷新，副屏立刻多出该章节的专属 Tab 页。

3.5 核心 5：一键复制章节为 Claude 上下文 (One-Click Context Bridge)

- **痛点**：副屏看 Spec 再方便，最终还是要切回终端给 Claude 喂上下文。手动复制粘贴章节内容既慢又容易遗漏关键段落。
- **解法**：在左侧切片树每个节点旁提供复制按钮，一键将该章节的完整 Markdown 内容复制到系统剪贴板。
- **Web 端呈现**：点击后 Toast 提示”已复制”，切回终端 `Ctrl+V` 即可将精准上下文粘贴给 Claude，真正打通副屏与终端主战场。

⚙️ 4. 技术栈选型与系统架构

4.1 技术栈推荐：Tauri 2.0 + React

为了追求极速开发、UI 顶尖质感，且不跟你的开发环境抢夺宝贵的内存和 CPU 资源，拒绝使用动辄数百 MB 的 Electron，全面采用 **Tauri 2.0**：

- **外壳框架**：Tauri 2.0（核心包仅十几 MB，常驻内存仅 20MB 左右，极其轻量）。
- **前端界面**：Vite + React + Tailwind CSS + Shadcn UI（组件美观且开发高效）。
- **核心组件**：`react-diff-viewer`（红绿比对墙内核）、`marked` 或 `react-markdown`（AST 分裂阅读）。
- **底层能力 (Rust)**：使用原生 Rust 的 `notify` 库进行高并发本地文件监听，利用系统原生 API 实现拖拽目录导入与突破浏览器沙盒读取类 Unix 根目录日志。

📅 5. 5阶段精简演进路线图 (Evolution Roadmap)

聚焦”先能存 → 再能读 → 最后能比”，每一步都有明确的使用价值，拒绝画大饼：

🏃‍♂️ 第一阶段：基础能力建设

- **Sprint 1：命令行追加打捞 (cs-save)**：编写 `cs-save` CLI 脚本，打通读取 Claude Code 会话 JSONL 日志通道，一键将最新对话格式化为 Markdown 并追加进 `spec/SPEC.md` 末尾的 Scratchpad 章节。**最早落地，确保开发工具期间的珍贵对话零丢失。**
- **Sprint 2：虚拟切片树 + 章节快速搜索**：实现拖入项目，把 `SPEC.md` 按 H2/H3 在前端分裂成目录树和 Tab 看板。同时内置模糊搜索，输入关键词一键跳转到对应章节。**解决大文档阅读困难。**
- **Sprint 3：暂存区 Diff 拦截与决策墙**：引入 Diff 组件，比对主文档与暂存文件，并排渲染方案对比卡片墙。用户手动输入最终选择原因后归档为 ADR。**阻止 AI 盲目改代码。**

🏃‍♂️ 第二阶段：深度纵向演进

- **Sprint 4：需求变更时光机**：剥离代码提交，通过过滤 Git Log，在副屏提供一条纯粹的 Spec 演进时间轴，拉动即可**回溯历史任意节点的业务条款与决策原因**。
- **Sprint 5：一键复制章节为 Claude 上下文**：在切片树每个节点旁提供复制按钮，一键将该章节内容复制到剪贴板，切回终端粘贴给 Claude 即可提供精准上下文。**打通副屏与终端主战场的桥梁。**
